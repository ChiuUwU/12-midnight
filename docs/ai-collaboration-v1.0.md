# Codex 与 Claude 协作开发规范 v1.0

> 本文档约束项目后续的多 AI 协作方式。目标是让每次修改可定位、可审查、可回滚，并减少重复阅读与重复实现。

## 1. 基本原则

1. 一个任务只有一个实现负责人，另一方负责审查或承接后续任务。
2. 两个 AI 不同时修改同一文件，尤其是 `web/app.js`、`server.js` 和 `functions/api/rooms/[[path]].js`。
3. 所有开发从明确的 Git 提交开始，以新的 Git 提交结束；禁止长期依赖无法追溯的未提交状态。
4. 规则文档是需求依据，代码和交接文档不得各自维护互相冲突的规则副本。
5. Cloudflare 只从测试通过的集成分支或 `main` 发布，并且一次发布只有一个负责人。

## 2. 任务分工

每个任务开始前明确以下角色：

- **实现负责人**：编写代码、测试和本轮交接记录。
- **审查负责人**：检查规则偏差、回归风险、测试缺口和部署影响。
- **集成负责人**：处理合并、版本号、数据库迁移和线上发布。小任务中可由实现负责人兼任。

分工按任务决定，不永久绑定某个 AI。推荐模式是“一方实现，另一方审查”，而不是双方同时实现同一功能。

## 3. 分支与工作区

- Codex 分支使用 `codex/<task-name>`。
- Claude 分支使用 `claude/<task-name>`。
- 并行开发必须使用不同 Git worktree，不能共享一个含未提交改动的工作目录。
- 只有修改文件完全不重叠时才允许并行开发。
- 若任务会同时触及核心单文件 SPA 或两套后端，改为串行开发。

开始任务前必须记录基准提交：

```text
Base commit: <commit hash>
Branch: <branch name>
Owner: Codex | Claude
Scope: <本轮唯一目标>
Owned files: <允许修改的文件>
```

## 4. 分层提交

不同性质的修改必须拆分提交：

1. 规则与产品文档。
2. 图片、音频等静态资源。
3. 角色和版型数据。
4. 核心规则与 API 实现。
5. UI 与交互。
6. 测试。
7. 数据库迁移与部署配置。

一个提交应只表达一个可以独立审查和回滚的意图。不要把新版型、无关统计功能、文档重写和部署修复放进同一个提交。

## 5. 文件所有权

任务进行期间，实现负责人临时拥有其声明的文件。另一方不得直接修改这些文件，除非：

1. 实现负责人已经提交并交接；或
2. 双方明确重新分配任务，并记录新的基准提交。

遇到已有未提交修改时：

- 不覆盖、不还原、不格式化他人的改动。
- 先确认这些改动的负责人和目标。
- 优先要求原负责人创建检查点提交。
- 必须接手时，先建立保留原状态的分支或提交，再继续修改。

## 6. 交接格式

每轮结束只需要更新简短交接记录，不重复复制全部项目背景：

```text
基准提交：
当前分支：
实现负责人：
本轮目标：
已完成：
修改文件：
测试命令与结果：
未完成或已知风险：
数据库/部署影响：
建议下一步：
下一位暂勿修改：
```

只有长期有效的信息进入 `CLAUDE.md`。具体任务进度写入 `docs/handoff.md`，完整规则写入对应版型或产品规则文档。

## 7. 审查与完成标准

实现负责人提交后，审查负责人基于提交差异进行检查，不重新实现同一功能。审查至少覆盖：

- 是否符合已确认规则。
- 本地模式与 Cloudflare Pages Function 是否一致。
- 非法官玩家是否可能获取秘密身份或夜间行动。
- 撤回、刷新、重复提交和角色死亡后的状态是否正确。
- 手机端布局和主要流程是否可用。
- 新功能是否有针对性的自动化测试。
- D1 字段或表结构变更是否有迁移说明。

完成标准：代码已提交、测试通过、交接已更新、线上影响已说明。仅“页面上看起来能用”不算完成。

## 8. 发布规则

1. 发布前确认工作区干净，并记录待发布提交哈希。
2. 运行自动化测试和语法检查。
3. 数据库迁移先执行并验证，再发布依赖新结构的代码。
4. 由唯一发布负责人触发 GitHub/Cloudflare 部署。
5. 发布后验证首页、创建房间、发牌、法官流程和本轮新增功能。
6. 将线上版本对应的提交哈希记录到交接文档；出现问题时按提交回滚。

## 9. Token 与时间优化

- 新会话优先读取 `CLAUDE.md`、当前任务规则文档和 `docs/handoff.md`，不要重放完整聊天历史。
- 提示词直接给出任务、基准提交、允许修改的文件和验收条件。
- 审查时提供提交哈希或 Git diff，不粘贴整个文件。
- 规则只维护一份权威版本，其他文档通过链接引用。
- 每完成一个小闭环就提交，避免积累数百行无法区分来源的改动。
- 搜索和读取按需进行，先看变更文件，不默认扫描整个仓库。
- 失败日志只保留错误核心、执行命令和必要上下文。

推荐的新任务提示词：

```text
先阅读 CLAUDE.md、docs/handoff.md 和本任务规则文档。
基准提交为 <hash>，在 <branch> 上工作。
本轮只完成 <goal>，只允许修改 <files>。
不要覆盖其他 AI 的未提交改动。
完成后运行 <tests>，提交代码，并按交接模板记录结果。
```

## 10. 后续架构协作方向

当前 `web/app.js` 和两套后端实现是并行开发的主要冲突点。功能稳定后，应逐步拆分：

- 版型与角色数据。
- 夜间步骤生成和通用结算规则。
- 页面渲染与事件处理。
- 本地 Node 与 Cloudflare 共用的纯规则函数。
- 每个版型独立的规则测试夹具。

拆分本身也按小步提交进行，不与新版型开发同时做大规模重构。模块边界稳定后，两个 AI 才适合真正并行开发不同版型或不同层级。
